# 08. 구현상세 — 테스트 신뢰성과 검증기 자기보호

## 1. Test Weakening Detector

### 탐지 대상

```text
- test file 삭제
- test case 삭제
- assert 삭제
- assert 조건 완화
- expected 값 완화
- exception expectation 제거
- skip/only/ignore 추가
- timeout/retry 증가
- snapshot 대량 변경
- mock이 실제 검증을 대체
```

### 구현 방식

1차는 텍스트 diff + 언어별 패턴으로 충분하다.

```text
old: expect(status).toBe(403)
new: expect(status).not.toBe(500)
=> 약화 의심
```

2차부터 AST 어댑터를 붙여 assert 호출의 operator, expected literal, matcher 종류를 비교한다.

## 2. Regression Corpus

```text
corpus/
  ai_mistakes/
  test_weakening/
  secret/
  dependency/
  validator_bypass/
  hardcode/
  schema/
  migration/
```

각 case는 다음 구조를 가진다.

```yaml
case_id: test-weakening-001
rule_id: test.assertion_weakened
should_fail: true
input_before: before.patch
input_after: after.patch
expected_diagnostic:
  severity: block
  contains: "assert가 약화"
```

## 3. Validator Self-Guard

검증기/policy/CI 변경은 일반 구현 작업과 분리한다.

BLOCK 기준:

```text
- validator 삭제
- rule severity 낮춤
- allowlist 무기한 추가
- suppression expiry 없음
- CI에서 검증 command 제거
- validator fixture 없이 새 rule 추가
```

## 4. Validator Quality Gate

새 validator에는 반드시 있어야 한다.

```text
- positive fixture: 잡아야 할 샘플
- negative fixture: 잡으면 안 되는 샘플
- edge fixture: 경계 사례
- golden diagnostic: 예상 rule_id/severity/location
- runtime budget: 너무 느리지 않은지
```

## 5. Mutation/Fuzz 적용 기준

전면 적용하지 않는다. 다음 영역만 장기적으로 적용한다.

```text
- test weakening detector
- validator selfguard
- policy/gate engine
- parser/serializer/file format
- CLI command parser
- compiler/IR 관련 core
```
