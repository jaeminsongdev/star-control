# 10. 구현상세 — 보안, 공급망, 신뢰성

## 1. 보안은 SAST-lite부터

처음부터 CodeQL급 정밀 taint를 만들지 않는다. 우선 위험도가 높고 구현 쉬운 패턴을 잡는다.

```text
- secret pattern/entropy
- private key / webhook / DB URL
- shell exec
- raw SQL string concat
- eval/dynamic import
- path traversal 의심
- TLS verify disabled
- non-crypto random for token
- auth/session/token 경로 변경 감시
```

## 2. Dependency/Supply Chain

### 우선 구현

```text
1. manifest/lockfile diff
2. new dependency approval
3. license policy
4. dependency purpose summary
5. SBOM-lite
6. OSV matcher
```

### SBOM-lite 필드

```json
{
  "component": "serde",
  "version": "1.0.0",
  "ecosystem": "crates.io",
  "direct": true,
  "license": "MIT OR Apache-2.0",
  "source_manifest": "Cargo.toml",
  "hash": "sha256:..."
}
```

## 3. OSV Matcher

취약점 데이터는 외부를 참조해야 한다. 자체 구현 대상은 다음이다.

```text
- ecosystem/name/version normalization
- version range matching
- advisory severity mapping
- affected/fixed 판단
- dependency graph와 연결
- review pack에 위험 요약 출력
```

## 4. VEX는 조건부

SBOM/OSV 오탐이 많아질 때만 OpenVEX 스타일 status를 붙인다.

```text
- affected
- not_affected
- fixed
- under_investigation
```

## 5. 결정성/환경의존성

AI가 만든 테스트와 검증기가 흔들리지 않게 한다.

```text
- 현재 시간 직접 사용 금지: clock wrapper
- 랜덤 직접 사용 금지: seeded random/fake random
- 네트워크 기본 금지: sandbox profile
- cwd/absolute path 의존 금지
- timezone/locale 고정
- sleep 기반 테스트 경고
```

## 6. Release Integrity Subset

공개 배포 전 구현한다.

```text
- source commit hash
- build command
- environment fingerprint
- dependency snapshot
- artifact hash
- SBOM-lite link
- release note consistency
```

TUF/in-toto/SLSA full chain은 자동 업데이트/공개 패키지 배포가 본격화될 때 subset에서 확장한다.
