# 09. 구현상세 — 정적분석, 아키텍처, 계약

## 1. 언어 어댑터 전략

처음부터 모든 언어의 완성형 parser/type checker를 만들지 않는다.

```text
Level 0: 파일/확장자/경로 분류
Level 1: import/export/function/test/assert 추출
Level 2: symbol table/public API index
Level 3: module dependency graph
Level 4: 간단한 type/signature diff
Level 5: 제한적 CFG/DFG/taint
```

첫 언어는 실제 구현 비중이 높은 언어 하나로 시작한다. Rust 기반 CLI라면 Rust adapter부터, 웹 GUI가 먼저라면 TypeScript adapter부터 시작한다.


## 3. Architecture Boundary

정책 예시:

```yaml
layers:
  domain:
    may_depend_on: []
  application:
    may_depend_on: [domain]
  infrastructure:
    may_depend_on: [domain, application]
  ui:
    may_depend_on: [application]

forbidden_imports:
  - from: "src/domain/**"
    to: "src/infra/**"
```

## 4. Contract Validators

장기 구현 가치가 높은 계약 검증:

```text
- public API signature diff
- CLI command/flag diff
- config schema diff
- error code registry diff
- DB migration safety
- serialization golden fixture compatibility
```

## 5. 하드코딩/가짜구현 탐지

탐지 예:

```text
- 함수가 입력을 사용하지 않고 상수 반환
- test file 이름/fixture 이름으로 분기
- catch 후 무시
- TODO/HACK가 production path에 남음
- fake success return
- not implemented but pass
```

## 6. 제외/유보

```text
- full type checker: 유보
- full formatter: 유보
- full interprocedural dataflow: 보안 규모가 커진 뒤
- formal proof: compiler/sandbox/permission core에만 장기 연구
```
