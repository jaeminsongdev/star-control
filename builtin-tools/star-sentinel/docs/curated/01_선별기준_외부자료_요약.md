# 01. 선별기준과 외부자료 조사 요약

## 1. 선별 원칙

v4에서는 기능을 많이 넣는 것이 목표가 아니다. 다음 조건을 만족하는 기능만 장기 구현 대상으로 남겼다.

| 기준 | 설명 |
|---|---|
| 최종검토 절약성 | 사용자가 diff 전체를 읽지 않고 승인/반려할 수 있게 하는가 |
| AI 사고 방지성 | 클라우드 AI가 흔히 하는 범위이탈, 테스트약화, 하드코딩, 검증우회를 잡는가 |
| 일반 개발 품질성 | AI 전용이 아니라 일반 코드 품질/보안/계약 안정성에도 도움이 되는가 |
| 재사용성 | One, Star-Control, RAG, GUI, DB, CLI 등 여러 프로젝트에 반복 사용 가능한가 |
| 구현비 대비 효용 | 1인 개발자가 AI로 구현시켜도 유지 가능한 범위인가 |
| 오탐 관리 가능성 | 예외/만료/baseline/fixture로 운영 가능한가 |

## 2. 외부자료에서 가져온 구현 관점

- **SARIF 2.1.0** (OASIS): 정적분석 결과 교환 포맷. 자체 diagnostic/report 모델의 구조 참고.  
  URL: https://www.oasis-open.org/standard/sarif-v2-1-0/
- **CodeQL** (GitHub): 코드를 DB처럼 모델링하고 AST/CFG/DFG를 질의하는 구조 참고. 도구 자체를 붙이지 않고 분석 DB 개념만 참조.  
  URL: https://codeql.github.com/docs/codeql-overview/about-codeql/
- **Tree-sitter** (Tree-sitter): 증분 파싱/언어 어댑터 구조 참고. 초기는 경량 파서, 장기는 언어별 어댑터.  
  URL: https://tree-sitter.github.io/
- **Semgrep rules** (Semgrep): 패턴 룰 + 메타변수 + dataflow 룰 작성 방식 참고. 자체 룰 DSL 설계에 반영.  
  URL: https://docs.semgrep.dev/writing-rules/overview
- **OPA** (Open Policy Agent): Policy-as-Code와 중앙 의사결정 엔진 개념 참고.  
  URL: https://openpolicyagent.org/docs
- **Cedar** (Cedar Policy): RBAC/ABAC 권한 정책 언어 구조 참고. agent/tool capability 정책 설계에 반영.  
  URL: https://cedarpolicy.com/
- **NIST SSDF SP 800-218** (NIST): 보안 소프트웨어 개발 실천 항목. secure development 요구사항의 상위 기준.  
  URL: https://csrc.nist.gov/pubs/sp/800/218/final
- **OWASP ASVS** (OWASP): 웹/서버 보안 요구사항 분류 기준. 모든 보안 rule을 다 구현하지 않고 위험경로 중심으로 선별.  
  URL: https://owasp.org/www-project-application-security-verification-standard/
- **OWASP Top 10 2025** (OWASP): 웹 애플리케이션 주요 보안 위험 분류.  
  URL: https://owasp.org/www-project-top-ten/
- **OWASP LLM Top 10 2025** (OWASP GenAI Security): AI/RAG/tool 호출 계층의 prompt injection, excessive agency, sensitive disclosure 위험 분류.  
  URL: https://genai.owasp.org/llm-top-10/
- **CycloneDX** (OWASP/Ecma): SBOM/BOM 객체 모델. 자체 SBOM-lite 설계 참고.  
  URL: https://cyclonedx.org/specification/overview/
- **SPDX** (Linux Foundation): SBOM, 라이선스, AI/data/security reference 표현 참고.  
  URL: https://spdx.dev/
- **OSV Schema** (OpenSSF): 취약점이 패키지 버전/커밋에 매핑되는 스키마. 자체 vulnerability matcher 데이터 모델 참고.  
  URL: https://github.com/ossf/osv-schema
- **OpenVEX** (OpenSSF): 취약점 영향 여부/VEX 상태 표현. 장기 공급망 검증의 조건부 기능.  
  URL: https://openssf.org/projects/openvex/
- **SLSA v1.2** (SLSA): 공급망 수준, provenance, attestation 참고. 개인 개발자는 release integrity subset만 우선.  
  URL: https://slsa.dev/spec/v1.2/
- **in-toto** (in-toto): 빌드/배포 단계가 계획대로 수행됐는지 metadata로 검증하는 구조 참고. 전체 도입은 공개 배포 시점.  
  URL: https://in-toto.io/
- **TUF** (The Update Framework): 업데이트 채널 보호. 자동 업데이트 기능이 생길 때 조건부 도입.  
  URL: https://theupdateframework.io/
- **OpenTelemetry** (CNCF): trace/metric/log와 context propagation 개념. run/task/validation ledger와 연결.  
  URL: https://opentelemetry.io/docs/
- **Stryker Mutation Testing** (Stryker): 테스트가 결함을 실제로 잡는지 확인하는 mutation testing 개념.  
  URL: https://stryker-mutator.io/docs/
- **libFuzzer** (LLVM): coverage-guided fuzzing. parser/serializer/compiler/format 처리에서 장기 필요.  
  URL: https://llvm.org/docs/LibFuzzer.html
- **WCAG 2.2** (W3C): GUI/Web을 만들 때 접근성 검증 기준. 모든 프로젝트의 필수는 아님.  
  URL: https://www.w3.org/WAI/standards-guidelines/wcag/

## 3. 외부 도구를 그대로 쓰지 않는다는 의미

이 설계에서 “커스텀 구현”은 다음을 뜻한다.

```text
- CodeQL/Semgrep/Sonar/Snyk/Trivy 같은 범용 도구를 파이프라인에 붙여서 끝내지 않는다.
- 다만 SARIF, OPA, CodeQL, Semgrep, CycloneDX, SPDX, SLSA 등의 공개 표준/아키텍처 개념은 참고한다.
- 취약점 DB처럼 외부 데이터가 필요한 영역은 데이터 소스만 참조하고, matcher/정책/리포트/승인 게이트는 자체 구현한다.
```

## 4. v4에서 의도적으로 줄인 영역

```text
- 모든 언어 완성형 타입체커
- 모든 언어 완성형 formatter
- 초기 full CFG/DFG/taint engine
- full formal verification
- full TUF/in-toto/SLSA release chain
- 전면 GUI/e2e/a11y/visual regression
- 전면 privacy data-flow
- 광범위한 package reputation scoring
```

이들은 “영원히 불필요”가 아니라, 현재 사용자의 개발 상황에서는 **조건부/후순위 subset만 구현하는 것이 합리적**이다.
