# 02. v3 감산 재검토 결과

## 1. v3의 문제

v3는 기능 누락 방지를 목적으로 405개 기능까지 확장했다. 설계 축 자체는 넓었지만, 실제 1인 개발자가 AI를 통해 구현하기에는 다음 문제가 있었다.

```text
1. 기능 수가 많아 우선순위가 흐려짐
2. full formal verification, full taint, full TUF 등 연구/엔터프라이즈급 기능이 초기에 섞임
3. GUI/Web/AI/RAG/DB/릴리즈 기능이 조건부인데도 동등한 필수 기능처럼 보임
4. “진짜 구현해야 할 P0/P1 validator”가 묻힘
5. 최종검토 비용 절감이라는 본래 목적보다 정적분석 백과사전이 됨
```

## 2. v4의 감산 기준

남긴 기능은 아래 질문에 “예”라고 답할 수 있어야 한다.

```text
- AI가 구현한 패치를 최종 승인하기 전에 실제 사고를 줄이는가?
- 1인 개발자의 검토 시간을 줄이는가?
- 여러 프로젝트에 재사용 가능한가?
- 커스텀 구현 가치가 외부 도구를 붙이는 것보다 큰가?
- 과도한 오탐/유지비 없이 운영할 수 있는가?
- 장기적으로 네 개발 생태계의 표준 브레이크가 될 수 있는가?
```

## 3. 유지한 영역

```text
- repo/diff/policy/diagnostic/ledger 기반
- sentinel task / scope / risk path / capability / AI claim 검증
- test weakening / validator bypass / regression corpus
- hardcode/fake implementation / hallucinated API 탐지
- architecture boundary / public API / schema / DB / serialization 계약
- secret / security-lite / dependency / license / SBOM-lite / OSV matcher
- determinism / sandbox runner / review pack / approval gate
- validator self-protection / allowlist expiry / fixture 의무화
```

## 4. 조건부로 내린 영역

| 영역 | v4 판단 |
|---|---|
| GUI/Web/E2E/a11y | GUI 제품을 만들 때만 subset 구현 |
| AI/RAG prompt/tool 검증 | AI 관제/RAG 기능이 실제 구현될 때 구현 |
| DB migration | DB를 도입하는 프로젝트부터 구현 |
| fuzz/property/mutation | parser/serializer/compiler/validator core에만 적용 |
| release provenance/TUF | 오픈소스/자동 업데이트 배포 직전 구현 |
| 정밀 taint/DFG | 서버/외부입력 계층이 커진 뒤 구현 |
| full formal verification | 일반 도구가 아니라 compiler/sandbox/permission core 일부만 장기 연구 |

## 5. 제외한 영역

```text
- AI 작성 코드 탐지기
- 코드 미학 점수화

- 과도한 자동 리팩터링 제안기
- 모든 플랫폼/브라우저 조합 전면 매트릭스
- 모든 의존성 reputation scoring 전면 구현
```
