# 11. 조건부/후순위 도구 판단표

## 1. 구현할 가치가 있지만 조건부인 것

| 기능 | 구현 트리거 | v4 판단 |
|---|---|---|
| GUI E2E | GUI/웹 제품이 실제로 생김 | 핵심 flow만 구현 |
| a11y/WCAG | GUI를 공개 배포하거나 사용자 대상 UI가 생김 | smoke + 핵심 기준만 |
| visual regression | UI 스타일 안정성이 중요해짐 | 주요 화면 snapshot만 |
| i18n 검증 | 다국어 UI/문서가 생김 | key 누락/unused부터 |
| DB migration safety | DB schema migration이 생김 | destructive operation 우선 |
| property/fuzz | parser/serializer/format/compiler 구현 | 해당 모듈만 |
| mutation testing | core validator 품질 검증 | validator/policy core만 |
| 정밀 taint | 외부 입력 서버가 커짐 | source/sink subset부터 |
| VEX | 취약점 오탐이 많아짐 | OSV 이후 |
| release provenance | 공개 배포/패키지 배포 | artifact hash부터 |
| TUF | 자동 업데이트 채널 | 그때 구현 |
| in-toto | 공급망 단계가 복잡해짐 | 공개 배포 후 |
| AI/RAG prompt 검증 | RAG/tool agent 구현 | prompt injection/tool permission subset |
| privacy flow | 사용자 파일/개인정보/RAG 처리 | log/export/external call부터 |
| compatibility matrix | 실제 지원 환경이 늘어남 | Windows+주요 toolchain부터 |

## 2. 구현하지 말 것 또는 연구로만 둘 것

| 기능 | 판단 | 이유 |
|---|---|---|
| AI 작성 코드 탐지기 | 제외 | AI가 썼는지가 아니라 맞는지가 중요 |
| 코드 미학 점수화 | 제외 | 검증 노이즈가 큼 |
| full multi-language type checker | 유보 | 구현비가 너무 큼, compiler/typechecker 영역 |
| full formatter | 유보 | 초반 효용 낮음 |
| full formal verification | 연구 | compiler/sandbox/permission 일부만 의미 있음 |
| 모든 플랫폼/브라우저 전면 matrix | 제외 | 1인 개발 유지비 과다 |
| 모든 의존성 reputation scoring | 유보 | 오탐/데이터 유지비 큼 |
| 모든 로그/telemetry full OTel 구현 | 유보 | 자체 ledger/event schema면 충분 |

## 3. 핵심 원칙
