# 확장 운영 기능

상위 범위와 공통 선정 기준은 [구현 대상 기능](README.md)에서 확인한다.


## D01. 여러 프로젝트·원격 Git·자료조사

하나의 목표가 여러 저장소와 최신 외부 사실에 걸릴 때도 같은 관제 원칙을 유지한다.

- 프로젝트별 정본, 역할, 선행 관계와 목표 graph
- 제공자 프로젝트와 소비자 프로젝트의 계약 변경 순서
- 프로젝트별 worktree, 상태, Context, 검사와 증거 분리
- 전체 목표의 통합 Checkpoint와 실패 시 부분 재개
- 원격 branch·PR·check 상태 조회와 로컬 revision 연결
- 승인 정책에 따른 push, PR 생성·수정, merge와 main 갱신
- Codex를 통한 자료조사 단계와 출처 URL, 확인 날짜, 적용 판단 기록
- 최신성이 필요한 주장은 근거 없이 확정하지 않고 재확인 대상으로 표시
- 로컬 작업 상태와 원격 서비스 상태를 명확히 분리

## D02. 비용·평가·규칙 개선

자동 배정과 검증 규칙이 실제로 1인 개발자의 시간을 줄이는지 측정하고 근거 있게 조정한다.

- 단계·실행별 시간, 사용량, 재시도, 대체, 재작업과 검토 시간 기록
- 잘못된 배정, 불필요한 대기와 버려진 작업 표시
- 품질, 안전, 사용자 수락과 완료 후 되돌림 결과 연결
- 실제 성공·실패 사례를 평가 Corpus로 축적
- 결정적 채점기를 우선하고 독립 Codex 검토는 필요한 표본에만 사용
- 실제 작업을 바꾸지 않는 shadow 비교로 새 규칙 시험
- model·생각 깊이·도구·검증 Profile 변경 전후 비교
- 정책과 평가 자료 version, 변경 이유와 회귀 결과 보존
- 자동 판단보다 사용자의 수동 선택과 예산 상한 우선

스스로 학습해 통제되지 않게 바뀌는 router는 만들지 않는다. 규칙 개선은 관측 자료와 검토 가능한 설정 변경으로만 한다.

## D03. Windows 배포와 제품 수명주기

개인 사용뿐 아니라 공개 배포를 전제로 설치부터 제거까지 관리한다.

- Windows 설치, 초기화, 진단, update와 uninstall
- Codex Plugin·MCP·Hook package와 신뢰·활성 상태 검사
- Controller 시작 방식과 종료·재시작·crash 복구
- 설정·상태·artifact 형식 version과 안전한 migration
- 공개용 `safe_default`와 개인용 `personal_auto` 예시
- 상태 export, 진단 Pack, 복구와 보존 자료 정리
- 배포 artifact file list, checksum, 출처와 release readiness
- update 실패 시 이전 version으로 되돌리는 절차

자체 browser UI와 반복 예약 기능은 포함하지 않는다. 사용자 입력은 Codex 앱, 제품 조작과 상태 확인은 터미널을 사용한다.
