# 07. 보안, 권한, 정책 설계

Star-Control은 여러 AI가 파일을 수정하고 명령을 실행하는 체계이므로, 보안/권한/승인 정책이 core 기능이어야 한다.

## 1. 기본 원칙

- 복구 어려운 작업은 자동 실행하지 않는다.
- 파일 삭제, 의존성 추가, 원격 push, git commit, 공개 API 변경은 사용자 승인 전 금지한다.
- 테스트 약화, 테스트 삭제, 검증 우회는 금지한다.
- 로컬 모델은 직접 repo 수정 권한을 기본적으로 갖지 않는다.
- 모든 worker는 WorkSpec의 allowed_scope 안에서만 작업한다.
- 검증하지 않은 작업을 통과로 표시하지 않는다.

## 2. 정책 파일 체계

```text
policies/
  command-policy.yaml
  approval-policy.yaml
  permission-policy.yaml
  sandbox-policy.yaml
  scope-policy.yaml
  risk-policy.yaml
  secret-policy.yaml
  budget-policy.yaml
```

## 3. Command Policy

`command-policy.yaml`이 원본이다. Codex의 `.rules`는 여기서 렌더링한다.

### forbidden 후보

- `rm -rf`
- `git reset --hard`
- `git clean`
- `git checkout -- .`
- `git restore .`
- `git push --force`
- `format`
- `diskpart`
- `Set-ExecutionPolicy`
- 테스트 삭제/스킵 자동화 명령

### prompt 후보

- `git commit`
- `git push`
- `git merge`
- `git rebase`
- `npm install`
- `pip install`
- `cargo add`
- Docker/Kubernetes/Cloud CLI
- 외부 네트워크 호출
- 시스템 패키지 설치

Codex `.rules`는 Starlark 형식이며 `codex execpolicy check`로 테스트할 수 있다. 출처: https://developers.openai.com/codex/rules

## 4. Approval Policy

항상 사용자 승인 필요:

- 새 의존성 추가
- 의존성 버전 변경
- 파일 삭제
- 공개 API 변경
- DB 스키마 변경
- 파일 포맷 변경
- 원격 저장소 push
- git commit
- 배포
- 외부 업로드
- 시스템 설정 변경

## 5. Permission Mode

Star-Control 내부 권한 모드:

| 모드 | 설명 |
|---|---|
| `read-only` | 파일 읽기만 가능, 설계/리뷰/요약 worker 기본 |
| `workspace-write` | WorkSpec 범위 안에서 파일 수정 가능 |
| `draft-only` | repo 수정 금지, draft.md만 작성 |
| `approval-required` | 승인 후에만 실행 가능 |
| `blocked` | 실행 금지 |

## 6. Sandbox Policy

Codex workspace-write는 기본적으로 명령 실행 중 네트워크 접근이 꺼져 있고, 필요 시 config로 켤 수 있다. Star-Control 기본값은 network off다. 출처: https://developers.openai.com/codex/agent-approvals-security

권장 기본값:

```yaml
sandbox:
  network_access: false
  writable_roots:
    - project_root
    - run_dir
  deny_paths:
    - "%USERPROFILE%/.ssh"
    - "%USERPROFILE%/.aws"
    - "%USERPROFILE%/.git-credentials"
```

## 7. Secret Guard

감지 대상:

- API key
- token
- password
- private key
- OAuth credential
- `.env` 값

정책:

- 비밀값 원문을 report에 쓰지 않는다.
- 위치와 위험만 요약한다.
- 외부 업로드 전 secret scan을 실행한다.

## 8. Scope Boundary

AI coding agent에서 자주 발생하는 문제는 작업 범위 이탈이다. 따라서 모든 WorkSpec에는 `allowed_scope`, `forbidden_actions`, `requires_approval`이 있어야 한다.

예:

```json
{
  "allowed_scope": ["src/stopwatch/**", "tests/stopwatch/**"],
  "forbidden_actions": ["새 의존성 추가", "파일 삭제", "git push"],
  "requires_approval": ["공개 API 변경"]
}
```

## 9. Checkpoint / Rollback

Claude Code는 checkpoint 기능을 제공하고, Codex/Cursor/GitHub/Jules류는 branch/worktree/PR review 흐름을 활용한다. Star-Control은 provider와 무관하게 VCS checkpoint를 자체 구현한다.

권장:

- 작업 전 `git status` snapshot
- 가능하면 별도 branch/worktree
- 변경 diff 저장
- 실패 시 변경 파일 목록과 rollback 방법 report

## 10. 인간 최종 권한

Star-Control은 자동 작업을 하더라도 merge/deploy/remote push의 최종 권한을 사용자에게 둔다.

원칙:

```text
AI는 작업을 제안/수행/검증할 수 있다.
최종 반영/배포/merge 권한은 사용자에게 있다.
```
