# 04. 핵심 스펙 설계

Star-Control에서 가장 먼저 고정해야 할 것은 스키마다. 스키마가 없으면 worker output이 매번 달라지고, Router Engine이 자동으로 다음 단계를 판단할 수 없다.

## 1. JobSpec

사용자 요청 하나를 표현한다.

필수 필드:

| 필드 | 설명 |
|---|---|
| `job_id` | `J-0001` 형식의 작업 ID |
| `project_root` | 작업 대상 프로젝트 루트 |
| `request_text` | 사용자 원 요청 |
| `created_at` | 생성 시각 |
| `entrypoint` | 처음 실행할 role |
| `state` | `REQUESTED` 등 현재 상태 |
| `user_constraints` | 사용자가 명시한 제약 |

상태값:

```text
REQUESTED
ROUTING
ROUTED
PLANNING
PLANNED
IMPLEMENTING
IMPLEMENTED
VALIDATING
VALIDATED
REVIEWING
REVIEWED
POLISHING
DONE
BLOCKED
FAILED
NEEDS_APPROVAL
CANCELLED
```

## 2. RouteSpec

라우터 판단 결과다.

필수 필드:

| 필드 | 설명 |
|---|---|
| `job_id` | 작업 ID |
| `summary` | 요청 요약 |
| `size` | SMALL / MEDIUM / LARGE / CRITICAL |
| `risk` | LOW / MEDIUM / HIGH / CRITICAL |
| `stages` | 실행할 stage 목록 |
| `assignments` | stage별 role/provider 배정 |
| `requires_user_approval` | 사용자 승인 필요 여부 |
| `approval_reasons` | 승인 필요 사유 |
| `workspecs` | stage별 WorkSpec 경로 또는 객체 |

## 3. WorkSpec

Worker에게 넘기는 작업 지시서다.

필수 필드:

| 필드 | 설명 |
|---|---|
| `job_id` | 작업 ID |
| `stage` | plan / implement / validate / review / polish 등 |
| `role` | worker role |
| `provider` | codex/local-ollama/claude 등 |
| `project_root` | 작업 디렉터리 |
| `goal` | 해당 worker의 목표 |
| `allowed_scope` | 수정 허용 범위 |
| `forbidden_actions` | 금지 행동 |
| `context_pack` | 요약 맥락 |
| `required_outputs` | 필수 산출물 |
| `validation_requirements` | 검증 요구사항 |

## 4. ReportSpec

Worker가 반환해야 하는 표준 보고다.

필수 필드:

| 필드 | 설명 |
|---|---|
| `job_id` | 작업 ID |
| `stage` | 수행 stage |
| `status` | DONE / FAILED / BLOCKED / NEEDS_APPROVAL / NEEDS_REVIEW |
| `changed_files` | 변경 파일 목록 |
| `commands_run` | 실행 명령 |
| `validation` | 검증 결과 |
| `risks` | 남은 위험 |
| `blocked_reason` | 막힌 이유 |
| `next_step` | 다음 단계 |
| `artifacts` | 출력물 경로 |

## 5. RunState

전체 작업 상태다.

필드:

| 필드 | 설명 |
|---|---|
| `job_id` | 작업 ID |
| `state` | 전체 상태 |
| `current_stage` | 현재 stage |
| `threads` | provider thread/session 목록 |
| `workers` | worker run 상태 |
| `artifacts` | 산출물 경로 |
| `next_action` | 다음 실행할 동작 |
| `budget` | 반복/시간/토큰 예산 |
| `history` | 주요 상태 전이 기록 |

## 6. Schema validation 원칙

- 모든 AI output은 schema validation을 통과해야 다음 단계로 넘어간다.
- schema validation 실패 시 worker에게 수정 요청하거나 `FAILED`로 기록한다.
- 사용자에게 보고하는 final report도 `ReportSpec` 기반으로 생성한다.
- provider-specific output은 adapter가 공통 schema로 normalize한다.

## 7. 스키마가 중요한 이유

AI worker는 말투, 형식, 상세도가 매번 달라질 수 있다. Star-Control은 사람이 보는 답변이 아니라 다음 단계 자동 실행을 위해 안정적인 data contract가 필요하다.

따라서 다음 원칙을 지킨다.

```text
대화형 응답은 보조 산출물이다.
자동화의 기준은 JSON ReportSpec이다.
```
