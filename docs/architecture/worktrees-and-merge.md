# 병렬 작업과 병합

## 목표

서로 겹치지 않는 단계를 동시에 실행하고, 각 결과를 검토 가능한 방식으로 하나의 최종 변경으로 합친다.

Star-Control은 새 버전 관리 시스템을 만들지 않는다. Git의 작업 복사본과 변경 기록 기능을 사용하고, 계획·충돌 확인·검사·통합 순서를 관리한다.

## 병렬 실행 조건

다음 조건을 만족할 때만 동시에 실행한다.

- 먼저 끝나야 하는 단계가 없음
- 수정 파일이나 책임 영역이 겹치지 않음
- 같은 전역 설정을 동시에 바꾸지 않음
- 결과를 독립적으로 검사할 수 있음
- 예상 비용과 동시 작업 한도 안에 있음
- 합치는 순서가 결과 의미를 바꾸지 않음

읽기 전용 조사는 같은 파일을 보더라도 병렬로 진행할 수 있다.

## 작업 복사본

수정 단계마다 필요하면 별도 Git worktree와 작업 브랜치를 만든다.

    star/<run-id>/<stage-id>

정확한 이름은 구현 계약에서 확정한다.

각 작업 복사본은 다음 정보를 가진다.

- 기준 변경점
- 담당 단계
- 수정 예상 범위
- Codex 작업 식별자
- 검사 상태
- 변경 기록 상태
- 병합 준비 상태

## 실행 흐름

1. 시작 전 기준 변경점을 고정한다.
2. 수정 범위가 겹치는지 미리 확인한다.
3. 각 단계에 별도 작업 복사본을 만든다.
4. 단계별 Codex 작업을 실행한다.
5. 각 단계에서 필요한 검사를 수행한다.
6. 검사를 통과한 결과만 병합 대기열에 넣는다.
7. 통합 전 다시 변경 범위와 충돌을 확인한다.
8. 정해진 순서로 통합 작업 복사본에 합친다.
9. 합친 상태에서 통합 검사를 실행한다.
10. 성공하면 기준 브랜치로 반영한다.

## GitHub와 비슷한 검토

로컬 단계도 검토 요청처럼 다음 정보를 제공한다.

- 변경 요약
- 관련 단계와 목표
- 바뀐 파일
- 검사 결과
- 남은 위험
- 기준 변경점
- 병합 가능 여부
- 충돌 여부

원격 저장소를 사용하는 경우 같은 정보를 실제 검토 요청에 연결할 수 있다.

## 충돌 처리

- 사용자 기존 변경을 자동으로 버리지 않는다.
- 충돌 파일과 관련 단계의 목적을 함께 보여준다.
- 단순하고 기계적인 충돌은 자동 해결할 수 있다.
- 의미 판단이 필요한 충돌은 Sol 단계로 보낸다.
- 해결 뒤 관련 검사와 통합 검사를 다시 실행한다.
- 안전하게 판단할 수 없으면 병합을 중단한다.

## 병합 순서

다음 기준으로 순서를 정한다.

- 기반 구조를 바꾸는 단계가 먼저
- 그 기반을 사용하는 단계가 나중
- 문서와 검사는 관련 구현과 함께
- 충돌 가능성이 큰 단계는 앞선 결과를 반영해 다시 실행

## 여러 프로젝트

여러 프로젝트에 걸친 목표는 프로젝트별 작업 복사본과 병합 상태를 따로 관리한다.

- 각 프로젝트에서 독립 검사를 통과해야 한다.
- 프로젝트 사이의 약속이 바뀌면 제공하는 쪽을 먼저 반영한다.
- 사용하는 쪽은 확정된 약속을 기준으로 작업한다.
- 전체 목표 완료 전 프로젝트 사이 연결 검사를 수행한다.

## 정리

- 완료된 임시 작업 복사본은 보관 정책에 따라 정리한다.
- 삭제는 설정된 승인 정책을 따른다.
- 작업 기록과 핵심 증거는 작업 복사본 삭제 뒤에도 남긴다.
