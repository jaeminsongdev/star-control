# 10. 검토 및 보강 필요 지점

이 문서는 지금까지 설계에서 빠지기 쉬운 부분과 보강해야 할 위험을 정리한다.

## 1. 부족할 수 있는 부분

### 1.1 Provider 기능 변화

AI coding agent 기능은 빠르게 변한다. Codex, Claude Code, Gemini CLI, Cursor, GitHub Copilot 모두 2026년에 기능이 계속 바뀌고 있다.

대응:

- `provider-features/*.features.yaml`에 `last_verified` 필드 추가
- 분기별 provider 기능 재조사
- adapter capability를 dynamic하게 비활성화 가능하게 설계

### 1.2 구조화 출력 안정성

AI가 항상 schema대로 출력하지 않을 수 있다.

대응:

- output repair step
- schema validation 실패 시 재질문
- 최종 실패 시 `FAILED_SCHEMA_VALIDATION`
- 가능하면 provider native structured output 사용

### 1.3 쓰레드 자동 재개

Provider마다 resume/fork/session semantics가 다르다.

대응:

- Star-Control RunState가 원본
- provider session id는 부속 정보
- resume 실패 시 fresh run + Context Pack으로 재개

### 1.4 Worktree 병렬성

동시에 같은 repo를 수정하면 충돌 위험이 있다.

대응:

- 초기 MVP는 순차 실행
- 병렬 worker는 git worktree 필수
- final merge 전 diff review 필수

### 1.5 명령 실행 안전성

provider가 shell command를 실행할 수 있으면 위험하다.

대응:

- command-policy.yaml 원본 관리
- provider별 command guard renderer
- shell wrapper 제한
- 승인 큐

### 1.6 로컬 모델 hallucination

로컬 모델은 작은 모델일수록 존재하지 않는 API/파일/설정을 만들 가능성이 높다.

대응:

- 직접 repo 수정 금지로 시작
- local output은 Cloud review 필수
- local worker role 제한

## 4. 설계상 중요한 판단

### 4.1 네이티브 기능은 선택적으로 사용

Provider native 기능이 있어도 Star-Control 내부 state를 원본으로 둔다.

예:

- Codex Goal을 쓰더라도 GoalSpec은 Star-Control에 저장
- Claude checkpoint를 쓰더라도 checkpoint metadata는 Star-Control RunState에 기록
- GitHub Copilot PR을 쓰더라도 PR metadata는 Star-Control artifact로 저장

### 4.2 Human final authority

자동화를 많이 하더라도 최종 merge/deploy/release 권한은 사용자에게 둔다.

### 4.3 Provider lock-in 방지

Codex profile, Claude plugin, Gemini extension, Cursor rule, GitHub skill은 모두 renderer 산출물이다. 원본은 Star-Control의 roles/skills/policies/schemas다.
