# 23. 부족 부분 검토 및 v2 보강 내역

## 1. 검토 기준

v1 패키지를 다음 기준으로 검토했다.

1. 문서만 보고 구현자가 시작할 수 있는가?
2. Provider 기능 차이를 Core에 누수하지 않는가?
3. 실패/재시도/승인/차단 흐름이 충분한가?
4. 스키마와 상태 전이가 구현 가능한가?
5. 테스트 전략이 있는가?
6. 보안/비용/관측성 설계가 있는가?
7. Codex 외 provider를 붙일 수 있는가?

---

## 2. v1에서 좋았던 부분

- Star-Control 중심 구조로 Codex를 1차 Adapter로 분리했다.
- 기능군을 capability로 분해했다.
- Core Spec, Policy, Role, Skill, Provider 폴더 구조를 잡았다.
- Codex용 profile/rules/skills 산출물 방향을 포함했다.
- 로컬 모델을 보조 worker로 제한했다.

---

## 4. 남은 과제

v2에도 아직 실제 구현 전 확인해야 할 사항이 있다.

1. 각 provider CLI의 실제 installed version 확인.
2. Codex `exec resume`의 옵션 지원 상태 실측.
3. Claude/Gemini/Cursor adapter는 공식 문서 확인 후 smoke test 필요.
4. Windows PowerShell escaping 규칙은 실제 command runner 구현 시 별도 테스트 필요.
5. 로컬 모델 OpenAI-compatible endpoint 설정은 Ollama/LM Studio/vLLM별로 실측 필요.
6. JSON Schema와 Markdown WorkSpec의 이중 관리 방식을 구현 중 확정해야 함.
7. UI는 MVP 이후로 미뤄야 함.

## 5. v2 결론

v2는 설계 문서만 보고도 아래 작업을 시작할 수 있는 수준을 목표로 보강했다.

```text
1. 폴더 구조 생성
2. Core schema 작성
3. Policy 작성
4. Fake provider 구현
5. Codex adapter 구현
6. star-control CLI MVP 구현
7. Codex smoke test 실행
8. Local draft worker 추가
```
