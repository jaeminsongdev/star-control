# ADR-0001: D0 최종 설계 기준

## 상태

채택 — 2026-07-11

후속 [ADR-0011](ADR-0011-Stable-rustfmt-Allowlisted-Clippy-Personal-Auto-경계.md)은 `rust_style_auto_fix`를 16번째 Profile로 추가했다. 아래 “15개 작업 유형”은 2026-07-11 당시 외부 입력 자료와 최초 선택 집합의 역사적 수이며 현재 최종 Profile 수는 16개다.

## 결정

새 Star-Control은 Windows용 Codex 전용 개발 제어 도구로 만든다. 사용자는 Codex 앱에서 목표를 입력하고, 제품은 terminal·Controller·MCP로 계획, 실행, 검사, 증거와 이어하기를 관리한다.

다음 경계는 확정한다.

- 로컬 AI, 다른 AI 제공자, OpenAI API 직접 호출, 브라우저 UI와 자체 예약 실행 기능을 만들지 않는다.
- 최종 runtime은 `star`, `star-controller`, `star-mcp` 세 실행 파일과 책임별 내부 Package로 나눈다.
- 외부 시스템 접근은 adapter에서만 수행하고 Controller가 실행 상태의 단일 writer가 된다.
- 23개 구현 기능은 공통 관제·검증 engine에 배치하고, 15개 작업 유형은 Profile로 조합한다.
- 현재 문서 경로를 정본으로 사용하고, `legacy/`는 로컬 읽기 전용 역사 자료로만 보존한다.

## RouteDecision 계약

RouteDecision은 다음 축을 섞지 않는다.

```text
model_role: sol | terra | luna
resolved_model: 실제 Codex 모델 ID
reasoning_effort: minimal | low | medium | high | xhigh
stage_mode: plan | execute | review
execution_mode: single | max | ultra
execution_realization: native | managed
capability_snapshot_ref: 실행 당시 지원 기능
```

`reasoning_effort`는 Codex 원시 설정 값이다. `max`와 `ultra`는 Star-Control이 한 단계의 실행 방식과 통합 방식을 표현하는 내부 값이며, 실행 시점 CapabilitySnapshot이 지원하지 않는 native 기능을 억지로 요청하지 않는다.

## 이유

- 생각 깊이, 계획 단계, 단일 강화 실행, 병렬 실행은 실패 방식과 비용이 달라 하나의 값으로 다루면 배정·복구·평가가 불명확해진다.
- Codex 제공 기능은 바뀔 수 있으므로 모델 이름과 native mode를 core 계약에 고정하지 않는다.
- 문서와 Package 경계를 먼저 고정해야 P1 Schema·상태·설정 계약이 여러 위치에서 중복되지 않는다.

## 결과

- P1은 이 계약을 `star-contracts`와 Schema fixture의 출발점으로 사용한다.
- 새로운 Codex 기능은 `star-adapter-codex`와 CapabilitySnapshot에서 흡수한다.
- 내부 실행 방식이 바뀌면 이 결정과 [라우팅 계약](../contracts/routing.md)을 함께 갱신한다.
