# ADR-0007: P0 하이브리드 저장소와 운영 정책

## 상태

채택 — 2026-07-12

## 맥락

[ADR-0006](ADR-0006-공통-개발-관리와-로컬-관리-DB-경계.md)은 Git 정본, 로컬 관리 DB, `.ai-runs`, 단일 Writer와 backend-neutral repository 경계를 고정했다. 구현에 들어가려면 한 사용자 공통 상태와 프로젝트별 대용량 projection을 어디에 둘지, 공유되지 않은 프로젝트의 ID를 어떻게 시작할지, 보안·보존·scan·판정·patch 기본값을 더 구체적으로 결정해야 한다.

하나의 DB만 쓰면 backup과 손상 범위가 모든 프로젝트로 퍼진다. 프로젝트마다 DB만 쓰면 프로젝트 발견, cross-project 관계와 여러 저장소에 걸친 작업 복구를 조정할 공통 위치가 없다. 따라서 두 책임을 분리하되 분산 transaction을 지원한다고 과장하지 않는 구조가 필요하다.

## 결정

### 하이브리드 저장소 topology

Controller는 다음 두 종류의 backend-neutral logical store를 연다.

- **global store**: Project directory projection, shared/local identity 상태, project-store locator, cross-project relation, multi-project coordination operation, global idempotency와 lifecycle summary를 저장한다.
- **project store**: 해당 ProjectId의 ProjectRevision, WorkspaceSnapshot, CanonicalSource, Symbol graph, ScanRun, Finding·Occurrence, local decision, ChangePlan·PatchSet, ValidationResult·GateDecision와 ArtifactRef index를 저장한다.

물리 위치는 사용자 local application data 아래 제품 전용 관리 root로 제한한다. project store directory 이름에는 ProjectId만 사용하며 project 이름, 사용자 이름과 source 절대 경로를 넣지 않는다. public 계약과 설정에는 backend 이름, DB filename과 connection string을 노출하지 않는다.

Controller 한 process가 global store와 모든 project store의 유일한 Writer다. 한 store 안의 event·projection·idempotency·revision 갱신은 원자적이지만, 여러 store를 하나의 ACID transaction으로 묶는다고 보장하지 않는다.

### 교차 store 작업

여러 store를 바꾸는 작업은 global store의 `CoordinatedOperation`과 각 project store의 participant receipt로 복구한다.

1. Controller가 global store에 operation ID, input fingerprint, expected version vector, participant ProjectId와 상태 `prepared`를 commit한다.
2. 각 project store transaction이 같은 operation ID와 payload fingerprint의 receipt를 결과와 함께 commit한다.
3. 모든 필수 participant가 확인되면 global operation을 `completed`로 commit한다.
4. crash 뒤 startup recovery는 receipt를 조회해 빠진 participant만 idempotent하게 재시도한다.
5. 결과를 확정할 수 없거나 precondition이 달라졌으면 `outcome_unknown` 또는 `blocked`로 두고 자동 pass·자동 rollback하지 않는다.

이미 commit된 participant를 숨은 역 transaction으로 되돌리지 않는다. 보상이 필요하면 별도 승인과 새 operation ID를 가진 명시적 compensating command를 만든다. backup·restore는 global generation과 관련 project generation을 한 manifest에 기록하고, 호환되지 않는 generation 조합으로 active pointer를 바꾸지 않는다.

### Project identity와 root binding

- 첫 local scan은 Controller가 발급한 local-only ProjectId로 시작할 수 있다.
- 여러 PC 공유, shared Baseline·Suppression, 장기 cross-project identity를 사용하려면 같은 ProjectId를 `.star-control/project.toml`에 선언해야 한다.
- local-only Project를 shared로 승격할 때 application service가 manifest PatchSet을 만들고 일반 preview·apply·validation 흐름을 거친다. 암묵적으로 파일을 만들지 않는다.
- root 절대 경로는 DB·event·artifact·fingerprint에 저장하지 않는다. 별도 root-binding store가 Windows current-user protection으로 암호화한 opaque locator만 저장하며 backup·export에서 제외한다.
- v1 DB 전체 암호화는 하지 않는다. current-user ACL과 persistence 전 redaction을 적용한다. 이는 관리자 권한 또는 이미 침해된 사용자 계정으로부터 DB를 보호한다는 주장이 아니다.

### lifecycle과 retention

- source-derived projection만 바꾸고 data loss가 없는 migration은 pre-migration backup 뒤 자동 실행할 수 있다.
- local decision 의미, redaction 의미 또는 data loss 가능성을 바꾸는 migration은 preview와 명시적 승인이 필요하다.
- 손상이나 미래 version을 발견하면 read-write handle을 닫고 read-only recovery로 전환할지 사용자에게 묻는다. 자동으로 다른 generation을 활성화하지 않는다.
- cleanup trigger는 startup과 수동 command뿐이다. 자체 scheduler는 만들지 않는다.
- 기본 보존은 complete scan 2세대, incomplete staging 7일, scan detail 90일, resolved finding과 local decision 180일, latest known-good와 pre-migration backup 최소 2개다. 완료 artifact는 90일, 해결된 실패 재현 artifact는 180일을 기본으로 한다.

### scan·decision·patch 기본 정책

- scan은 untracked 포함, ignored 제외, symlink 미추적, binary metadata-only를 기본으로 한다.
- 한 파일 16 MiB, 한 run 200,000개 파일·8 GiB, file 분석 동시성 4를 기본 상한으로 한다.
- Rule 오류와 persistence 전에 폐기해야 하는 secret·사용자 이름·raw 절대 경로·민감 literal을 만나면 원문과 hash를 저장하지 않고 ScanRun을 incomplete로 만든다. incomplete 결과는 auto-pass할 수 없다.
- shared Baseline·Suppression은 Git 선언, Disposition은 local decision이다. local 판단을 공유하려면 PatchSet으로 명시적으로 승격한다.
- Baseline은 complete scan 결과를 사람이 검토하고 명시적으로 생성할 때만 active가 된다.
- Suppression 기본 만료는 90일이다. 영구 suppression은 explicit permanent flag와 justification이 있어야 한다.
- PatchSet은 preview 뒤 명시적으로 apply한다. dirty workspace에서도 대상별 exact before hash가 일치하고 기존 변경과 겹치지 않을 때만 적용한다. 겹치면 별도 worktree를 사용하거나 block한다.
- apply 뒤 required validation을 자동 실행한다. exact reverse patch와 precondition이 모두 증명될 때만 자동 rollback하며, 그 외 partial 상태는 성공으로 숨기지 않는다.

### 원격 작업과 설치

- 공개 `safe_default`의 remote write는 항상 prompt다.
- `personal_auto`도 remote write는 기본 prompt이며 사용자가 대상 host·repository·action scope를 명시적으로 opt-in한 경우에만 그 범위에서 auto가 된다.
- P9의 주 배포 방식은 installer다. 구체 installer 기술은 구현 직전 공식 지원·서명·update 요구를 조사해 고른다.
- installer는 Controller current-user autostart를 눈에 띄게 설명하고 기본 활성화하되, 설치 중 해제와 설치 후 enable·disable·status를 제공한다. autostart는 Controller만 시작하며 작업을 예약하지 않는다.
- elevated executor는 실제 필수 use case와 별도 위협 모델이 승인되기 전까지 포함하지 않는다.

## 결과

- 한 project DB의 손상·backup·retention을 다른 project와 분리하면서 공통 발견과 cross-project 복구 상태를 유지한다.
- cross-store partial commit이 명시적인 상태와 복구 절차를 가지며 허위 원자성이나 허위 성공을 만들지 않는다.
- local-first onboarding과 Git 기반 공유 identity가 함께 가능하다.
- P0 구현자가 설정 기본값, 보안 경계, incomplete 처리, baseline·suppression과 dirty patch 동작을 추측하지 않아도 된다.

비용은 두 종류 store lifecycle, version vector와 coordination recovery를 구현·검증해야 한다는 점이다. 전체 DB 암호화를 선택하지 않았으므로 저장 가능한 값의 redaction과 Windows ACL 검증이 필수다.

## 선택하지 않은 대안

### 모든 프로젝트를 하나의 DB에 저장

검색은 단순하지만 손상·backup·migration·retention 범위가 모든 프로젝트에 결합되므로 선택하지 않았다.

### 프로젝트별 DB만 두고 global store를 만들지 않음

project discovery, cross-project relation과 partial operation 복구를 filesystem scan과 임시 상태에 맡기게 되므로 선택하지 않았다.

### public contract에 특정 DB backend를 노출

backend 교체가 CLI·MCP·Schema migration이 되는 결합을 만들므로 선택하지 않았다.

### dirty workspace의 변경을 자동 덮어쓰기

사용자 변경 유실과 잘못된 rollback 위험이 있으므로 exact hash·overlap 검사를 통과한 대상만 허용한다.
