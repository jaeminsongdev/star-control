# ADR-0002: 데이터 계약과 설정 정본

## 상태

채택 — 2026-07-11

후속 [ADR-0011](ADR-0011-Stable-rustfmt-Allowlisted-Clippy-Personal-Auto-경계.md)은 final Catalog에 `rust_style_auto_fix`를 추가했다. 아래 “15개 개발 작업 Profile”은 2026-07-11 최초 계약 집합의 역사적 수이며 현재 최종 작업 Profile 수는 16개다. 데이터 정본·derived state 경계는 바뀌지 않는다.

## 결정

Star-Control의 사람이 읽는 계약 의미는 `docs/contracts/`가 소유하고, 구현 단계의 기계 정본은 `star-contracts` Rust type과 version별 validation logic이 소유한다. JSON Schema, TOML reference, fixture index와 Markdown 목록은 이 기계 정본에서 생성한다.

다음 경계를 함께 고정한다.

- 저장·전달 계약마다 `schema_id`와 독립 `schema_version`을 둔다.
- EventEnvelope가 변경 이력의 근거이고 RunSnapshot은 재구축 가능한 조회 자료다.
- Controller만 상태의 writer이며 CLI와 MCP는 같은 typed application command를 local IPC로 보낸다.
- local IPC는 current-user Windows named pipe와 length-prefixed UTF-8 JSON을 사용한다.
- 설정은 typed layer merge, field별 merge strategy와 provenance를 거쳐 EffectiveConfig로 만든다.
- `safe_default`·`personal_auto` 정책 Profile과 15개 개발 작업 Profile을 별도 계약으로 둔다.
- Catalog는 실행 code가 아닌 Task·Tool·Check·작업 Profile·정책 Profile descriptor를 제공한다.
- raw shell string, secret 원문과 외부 protocol 원문을 core 계약에 넣지 않는다.
- Star-Control 설정은 Codex·운영체제·관리자 제한을 약화할 수 없다.
- 미래 version은 read-only inspection만 허용하고 migration 없이 쓰지 않는다.

## 정본 관계

```text
docs/contracts/*.md
  = 의미, 불변식, 책임 경계

star-contracts Rust type + validation
  = 기계 직렬화와 실행 검증 정본
       -> generated JSON Schema
       -> valid/invalid/version fixture
       -> generated reference 문서
```

Schema만으로 표현하기 어려운 graph, 상태 전이, permission과 cross-field 규칙은 `star-contracts`의 명시적 validator가 소유하며 같은 fixture로 검사한다. 사람이 작성한 Schema와 Rust type을 두 정본으로 병행하지 않는다.

## 이유

- 여러 Package, CLI, MCP, state와 report가 같은 type 의미를 공유해야 wire type 복제를 막을 수 있다.
- snapshot만 저장하면 실패와 승인 이유가 덮이고, event만 사용하면 조회와 이어하기 비용이 커진다.
- 설정 source마다 일반 덮어쓰기를 사용하면 하위 설정이 permission과 budget 제한을 실수로 넓힐 수 있다.
- MCP와 CLI에 별도 engine을 두면 같은 명령이 서로 다른 상태와 승인 결과를 만들 수 있다.
- version과 migration을 저장 뒤에 추가하면 공개 update·rollback에서 원본을 안전하게 보존하기 어렵다.

## 결과

- P1은 [데이터 계약 지도](../contracts/README.md)의 Inventory부터 type, Schema와 fixture를 생성한다.
- `star-config`는 StarConfig parse·merge·provenance와 Catalog resolution을 단독 소유한다.
- `star-state`는 event commit, snapshot reduce와 recovery를 소유한다.
- `star-ipc`는 transport와 message framing만 소유하고 application 의미를 복제하지 않는다.
- 새로운 계약이나 설정 key는 문서, type, version, fixture와 migration 판단을 한 변경으로 제출해야 한다.
