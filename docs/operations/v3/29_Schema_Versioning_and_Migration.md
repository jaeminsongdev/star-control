# 29. Schema Versioning과 Migration 전략

## 목적

Star-Control은 장기간 발전할 시스템이다. JobSpec, WorkSpec, ReportSpec, RunState가 바뀌면 기존 run artifact와 호환성이 깨질 수 있다. 따라서 처음부터 versioning과 migration을 설계한다.

## 버전 필드

모든 주요 JSON/YAML artifact에는 `schema_version`을 넣는다.

```json
{
  "schema_version": "1.0.0",
  "job_id": "J-0001"
}
```

## SemVer 규칙

| 변경 | 예시 | 버전 |
|---|---|---|
| patch | 설명 필드 추가, enum 설명 보강 | 1.0.1 |
| minor | optional field 추가 | 1.1.0 |
| major | required field 변경, enum 의미 변경 | 2.0.0 |

## 호환 정책

- Router Engine은 같은 major version만 직접 처리한다.
- minor version 차이는 기본값 주입으로 처리한다.
- major version 차이는 migration을 요구한다.

## Migration 파일

```text
migrations/
  001_job_1_0_to_1_1.py
  002_report_1_0_to_1_1.py
```

## Migration manifest

```yaml
from: "1.0.0"
to: "1.1.0"
artifacts:
  - job
  - route
  - report
steps:
  - add_field: schema_version
  - rename_field: commands_run -> evidence.commands
```

## Run artifact 보존

기존 run은 원본을 덮어쓰지 않는다.

```text
runs/J-0001/
  route.json
  migrations/
    route.v1.0.0.json
    route.v1.1.0.json
```

## Renderer 버전

Provider 산출물에도 원본 버전을 주석으로 넣는다.

```text
# Generated by Star-Control renderer
# source_schema_version: 1.1.0
# source_hash: abc123
```

## 구현 체크리스트

- [ ] 모든 schema에 `schema_version` 추가.
- [ ] loader가 major mismatch를 감지한다.
- [ ] migration command를 제공한다.
- [ ] renderer가 source version/hash를 출력한다.
- [ ] backward compatibility 테스트를 만든다.
